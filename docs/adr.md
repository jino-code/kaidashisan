# ADR-001: フレームワークにNext.jsを採用する

## ステータス
決定

## 背景と課題
- 買い出しさんのフロントエンド・バックエンドのフレームワークを選定する必要がある。
- 個人開発かつMVPとして素早く動くものを作ることを優先したい。

## 検討した選択肢
- Next.js
- Web API＋Reactなど

## 決定内容
Next.jsを採用する。

## 結果・影響
- 良い点
  - 別プロジェクトで開発経験があり、その経験をそのまま活かせる。
  - フロントエンドとバックエンドを一つのフレームワークで完結できる。
  - MVPとして素早く作れる。
- 悪い点・トレードオフ
  - Web API構成と比べてスケールしにくい。

# ADR-004: LINEとの連携設計

## ステータス
決定

## 背景と課題
- LINE Messaging APIには月ごとのメッセージ送信数の制限がある。
- Botからの返信設計と、家族へのリスト共有方法を決める必要がある。

## 検討した選択肢
**Botからの返信**
- メッセージ受信のたびに「追加しました」と返信する
- 返信なしでWebアプリ側にのみ反映する

**リスト共有**
- 固定URLで共有する
- 期限付きURLをLINEに送信する
- BotがLINEにリスト内容をテキストで送信する

## 決定内容
Botからの返信は原則行わず、Webアプリ側にのみ反映する。管理者がボタンを押すことで期限付きURL（24時間有効）をLINEグループに送信する。

## 結果・影響
- 良い点
  - メッセージ数の消費を少なくできる。
  - 無料枠で十分運用できる。
  - 期限付きURLのため、固定URLより安全。
- 悪い点・トレードオフ
  - 家族がLINEに送った後に何も反応がないため、追加されたか不安に思う可能性がある。
  - URLが流出した場合、24時間以内であれば第三者がアクセスできる。

# ADR-005: groupIdの登録をコピペで行う

## ステータス
決定

## 背景と課題
- 管理者がLINEグループとWebアプリを紐づけるための方法を決める必要がある。
- BotがLINEグループに参加した時にjoinイベントが発火してgroupIdを取得できるが、Webhookのデータにはどのユーザーアカウントのものかという情報が含まれていない。そのため、ユーザーとgroupIdを紐づける仕組みが別途必要になる。

## 検討した選択肢
- BotがLINEグループに参加した瞬間に自動でgroupIdを登録する
- BotがLINEグループに参加した時にgroupIdをグループに投稿し、管理者がアプリにコピペして登録する

## 決定内容
BotがLINEグループに参加した時にgroupIdをグループに投稿し、管理者がそのgroupIdをWebアプリにコピペして登録する。

## 結果・影響
- 良い点
  - ログイン中の管理者がgroupIdを入力することで、ユーザーとグループを確実に紐づけられる。
- 悪い点・トレードオフ
  - 管理者がgroupIdをコピペする手間が発生する。

# ADR-006: MVPでは管理者1人のみアプリにログインする

## ステータス
決定

## 背景と課題
- 複数の家族メンバーがアプリにログインして同じリストを共有する場合、ユーザーとリストの多対多関係を管理する中間テーブルが必要になる。
- MVPとして複雑さを最小限に抑えたい。

## 検討した選択肢
- 管理者1人のみアプリにログインする（家族は共有URLでリストを閲覧）
- 複数ユーザーがアプリにログインして同じリストを共有する
- 同じアカウントを家族で共有する

## 決定内容
MVPでは管理者1人のみアプリにログインする前提とする。家族はLINEで追加し、共有URLでリストを閲覧する。将来的には招待コードによる複数ユーザー対応を検討する。

## 結果・影響
- 良い点
  - 中間テーブルが不要でDB設計がシンプルになる。
  - MVPとして素早く作れる。
- 悪い点・トレードオフ
  - 複数ユーザーがアプリにログインする場合は中間テーブルの追加が必要。
  - 同じアカウントを共有する場合はセキュリティ上問題がある。

# ADR-007: PWA対応を採用する

## ステータス
決定

## 背景と課題
- 買い出しさんはスマホでの使用が主な想定のため、スマホアプリに近い体験を提供したい。
- ネイティブアプリの開発コストは避けたい。

## 検討した選択肢
- PWA対応のWebアプリ
- iOS/Androidのネイティブアプリ
- Webアプリのみ

## 決定内容
PWA対応を採用する。

## 結果・影響
- 良い点
  - ホーム画面に追加することでアプリに近い体験を提供できる。
  - ネイティブアプリの開発・審査コストが不要。
  - next-pwaを使えば比較的簡単に対応できる。
- 悪い点・トレードオフ
  - ネイティブアプリと比べてプッシュ通知などの一部機能に制限がある。
  - iOSではPWAのサポートが一部制限されている。

# ADR-009: データの削除は物理削除とする

## ステータス
決定

## 背景と課題
買い物リストのアイテムを削除する際、物理削除と論理削除のどちらを採用するか決める必要がある。

## 検討した選択肢
- 物理削除
- 論理削除

## 決定内容
物理削除を採用する。

## 結果・影響
- 良い点
  - DB設計がシンプルになる。
  - 削除済みデータのフィルタリングが不要。
- 悪い点・トレードオフ
  - 削除したデータを復元できない。
  - 将来的に削除履歴が必要になった場合は論理削除への移行が必要。
